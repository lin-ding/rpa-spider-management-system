# 流程管理接口调试文档

## 1. 文档说明

本文档基于当前后端代码中的实际实现整理，用于调试现有流程管理模块。

- 服务前缀：`http://localhost:8080/api`
- 模块前缀：`/process`
- 返回结构：统一使用 `Result<T>`

当前流程管理已经实现的能力：

- 流程分页查询
- 流程详情查询
- 流程新增
- 流程修改
- 流程删除
- 流程设计保存
- 流程编码唯一性校验
- 流程脚本测试占位接口

当前流程管理尚未实现的能力：

- 四个标准步骤 `采集 -> 解析 -> 加工 -> 落库` 的结构化建模
- 步骤级独立配置
- 真实脚本执行测试
- 独立的启用/禁用切换接口

## 2. 统一返回格式

成功返回示例：

```json
{
  "code": 200,
  "message": "操作成功",
  "data": {},
  "timestamp": 1712023200000
}
```

失败返回示例：

```json
{
  "code": 500,
  "message": "流程不存在",
  "data": null,
  "timestamp": 1712023200000
}
```

## 3. 流程对象字段说明

当前流程实体字段如下：

| 字段名 | 类型 | 说明 |
| --- | --- | --- |
| `id` | `Long` | 流程主键 ID |
| `processName` | `String` | 流程名称 |
| `processCode` | `String` | 流程编码，要求唯一 |
| `description` | `String` | 流程描述 |
| `processType` | `Integer` | 流程类型，当前约定：`1-数据采集`、`2-数据处理`、`3-自动化任务` |
| `scriptContent` | `String` | 流程脚本内容，当前为整段文本 |
| `status` | `Integer` | 状态：`0-停用`，`1-启用` |
| `createBy` | `Long` | 创建人 ID |
| `createTime` | `LocalDateTime` | 创建时间 |
| `updateTime` | `LocalDateTime` | 更新时间 |
| `deleted` | `Integer` | 逻辑删除标识，接口层一般不用传 |

## 4. 鉴权说明

以下接口需要管理员权限：

- `POST /process`
- `PUT /process/{id}`
- `DELETE /process/{id}`
- `PUT /process/{id}/design`

调试时需要在请求头中传入：

```http
Authorization: Bearer 你的token
```

获取 token 的登录接口：

- `POST /api/user/login`

登录请求示例：

```json
{
  "username": "admin",
  "password": "admin123"
}
```

## 5. 接口列表

### 5.1 分页查询流程列表

- 请求方式：`GET`
- 请求路径：`/api/process/list`
- 是否需要登录：否
- 是否需要管理员：否

请求参数：

| 参数名 | 位置 | 必填 | 类型 | 说明 |
| --- | --- | --- | --- | --- |
| `current` | query | 否 | `Integer` | 当前页码，默认 `1` |
| `size` | query | 否 | `Integer` | 每页条数，默认 `10` |
| `processName` | query | 否 | `String` | 按流程名称模糊查询 |
| `processCode` | query | 否 | `String` | 按流程编码模糊查询 |
| `status` | query | 否 | `Integer` | 按状态筛选 |
| `processType` | query | 否 | `Integer` | 按流程类型筛选 |

请求示例：

```http
GET http://localhost:8080/api/process/list?current=1&size=10&processName=股票&status=1
```

返回示例：

```json
{
  "code": 200,
  "message": "操作成功",
  "data": {
    "records": [
      {
        "id": 1,
        "processName": "股票数据采集流程",
        "processCode": "stock_collection",
        "description": "采集股票市场数据并存储到数据库",
        "processType": 1,
        "scriptContent": null,
        "status": 1,
        "createBy": 1,
        "createTime": "2026-03-26 23:00:00",
        "updateTime": "2026-03-26 23:00:00",
        "deleted": 0
      }
    ],
    "total": 1,
    "size": 10,
    "current": 1,
    "pages": 1
  },
  "timestamp": 1712023200000
}
```

### 5.3 新增流程

- 请求方式：`POST`
- 请求路径：`/api/process`
- 是否需要登录：是
- 是否需要管理员：是

请求头：

```http
Authorization: Bearer 你的token
Content-Type: application/json
```

请求体：

```json
{
  "processName": "基金数据采集流程",
  "processCode": "fund_collection",
  "description": "采集基金净值数据",
  "processType": 1,
  "scriptContent": "def execute() { return 'ok' }",
  "status": 1
}
```

说明：

- `processCode` 必须唯一
- `createBy` 由后端从当前登录用户自动写入
- `id`、`createTime`、`updateTime` 不需要前端传

成功返回：

```json
{
  "code": 200,
  "message": "操作成功",
  "data": null,
  "timestamp": 1712023200000
}
```

失败示例：

```json
{
  "code": 500,
  "message": "流程编码已存在",
  "data": null,
  "timestamp": 1712023200000
}
```

### 5.4 修改流程

- 请求方式：`PUT`
- 请求路径：`/api/process/{id}`
- 是否需要登录：是
- 是否需要管理员：是

请求头：

```http
Authorization: Bearer 你的token
Content-Type: application/json
```

路径参数：

| 参数名 | 必填 | 类型 | 说明 |
| --- | --- | --- | --- |
| `id` | 是 | `Long` | 流程主键 ID |

请求体示例：

```json
{
  "processName": "基金数据采集流程-更新",
  "processCode": "fund_collection",
  "description": "采集并处理基金净值数据",
  "processType": 2,
  "scriptContent": "def execute() { return 'updated' }",
  "status": 1
}
```

调试注意：

- 当前实现中建议修改时始终把 `processCode` 一并传回
- 如果流程不存在，会返回 `流程不存在`
- 如果修改后的编码与其他流程冲突，会返回 `流程编码已存在`

### 5.6 保存流程设计

- 请求方式：`PUT`
- 请求路径：`/api/process/{id}/design`
- 是否需要登录：是
- 是否需要管理员：是

说明：

- 当前接口与普通修改流程共用同一套保存逻辑
- 可以理解为“设计页保存”入口
- 目前并没有四步流程的独立结构化字段，实际保存的仍是 `RpaProcess`

请求示例：

```json
{
  "processName": "四步采集流程",
  "processCode": "four_step_process",
  "description": "流程设计页保存",
  "processType": 1,
  "scriptContent": "{\n  \"collect\": \"...\",\n  \"parse\": \"...\",\n  \"transform\": \"...\",\n  \"save\": \"...\"\n}",
  "status": 1
}
```

调试建议：

- 如果你前端已经做了四步设计器，可以先把四步内容整体序列化后塞进 `scriptContent`
- 这是当前后端最接近“流程设计保存”的调试方式

- 请求路径：`/api/process/test-script`
- 是否需要登录：否

请求体示例：

```json
{
  "processName": "测试流程",
  "processCode": "test_process",
```

### 5.8 校验流程编码唯一性

- 请求方式：`GET`
- 请求路径：`/api/process/checkProcessCode`
- 是否需要登录：否
- 是否需要管理员：否

请求参数：

| 参数名 | 位置 | 必填 | 类型 | 说明 |
| --- | --- | --- | --- | --- |
| `processCode` | query | 是 | `String` | 要校验的流程编码 |
| `id` | query | 否 | `Long` | 编辑场景下可传当前流程 ID，排除自身 |

请求示例：

```http
GET http://localhost:8080/api/process/checkProcessCode?processCode=stock_collection
```

返回示例：

```json
{
  "code": 200,
  "message": "操作成功",
  "data": false,
  "timestamp": 1712023200000
}
```

返回值说明：

- `true`：编码可用
- `false`：编码已存在

## 6. 推荐调试顺序

建议按下面顺序调试当前流程管理：

1. 调 `GET /api/process/list`，确认列表可返回
2. 调 `GET /api/process/{id}`，确认详情可查看
3. 调 `GET /api/process/checkProcessCode`，确认编码校验正常
4. 登录获取管理员 token
5. 调 `POST /api/process` 新增流程
6. 调 `PUT /api/process/{id}` 修改流程
7. 调 `PUT /api/process/{id}/design` 模拟设计页保存
8. 调 `POST /api/process/test-script` 验证脚本测试接口通路
9. 调 `DELETE /api/process/{id}` 删除测试数据

## 8. 当前后端调试注意事项

### 8.1 流程设计接口并不是真正的四步设计接口

当前后端只有一个 `scriptContent` 字段，所以前端如果要调“流程设计”，建议先将四步内容整体存成文本或 JSON 字符串后再提交。

### 8.2 更新接口建议传完整对象

当前更新逻辑不是字段级安全合并，调试时建议把：

- `processName`
- `processCode`
- `description`
- `processType`
- `scriptContent`
- `status`

全部一起传，避免出现字段被置空的问题。

### 8.3 脚本测试接口是占位实现

`/api/process/test-script` 目前不会真实执行 Groovy 或 Java 脚本，只会固定返回成功消息。
