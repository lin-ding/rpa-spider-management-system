# RPA 爬虫运营管理系统当前功能、接口与数据库说明

## 1. 项目现状概览

- 项目类型：Spring Boot 3 + MyBatis-Plus + MySQL 的后端服务。
- 服务地址：默认 `http://localhost:8080/api`
- 当前已接入的核心能力：
  - JWT 登录认证
  - 用户管理
  - 角色管理
  - 菜单管理
  - RPA 流程管理
  - 数据采集结果查询
  - 基于 AOP 的管理员权限校验
- 当前仍是占位/临时实现的模块：
  - 任务管理
  - 机器人管理
  - 执行监控

## 2. 当前已经有的功能

### 2.1 认证与安全

- 用户登录：校验用户名、BCrypt 密码、用户状态，成功后签发 JWT。
- 用户登出：将 token 加入黑名单。
- Token 校验：支持校验 token 是否有效。
- 当前登录人信息：可根据 token 获取用户基础资料。
- 个人资料修改：可修改真实姓名、邮箱、手机号。
- 个人密码修改：校验旧密码后更新为新密码。
- 管理员权限控制：标注 `@RequireAdmin` 的接口要求当前用户具有 `super_admin` 角色。

### 2.2 用户管理

- 用户分页查询
- 带角色信息的用户分页查询
- 用户详情查询
- 新增用户
- 修改用户
- 删除用户
- 重置密码
- 启用/禁用切换
- 用户名唯一性校验
- 按用户名关键字搜索用户

### 2.3 角色管理

- 角色分页查询
- 带用户数统计的角色分页查询
- 查询所有角色
- 查询角色详情
- 新增角色
- 修改角色
- 删除角色
- 角色编码唯一性校验

### 2.4 菜单管理

- 查询完整菜单树
- 查询菜单详情
- 新增菜单
- 修改菜单
- 删除菜单
- 菜单名称唯一性校验

### 2.5 RPA 流程管理

- 流程分页查询
- 流程详情查询
- 新增流程
- 修改流程
- 删除流程
- 保存流程设计
- 流程编码唯一性校验

### 2.6 数据采集结果

- 数据结果分页查询
- 数据详情查询

### 2.7 占位接口

- `/task/list`：返回空分页数据
- `/robot/list`：返回空分页数据
- `/robot/status-statistics`：返回全 0 统计
- `/execution/list`：返回空分页数据
- `/data/export`：返回“导出功能开发中”
- `/process/test-script`：固定返回“脚本测试成功”
- `/user/menu/tree`：认证模块中的菜单树目前返回空列表，尚未接真实菜单查询逻辑

## 3. 接口与作用汇总

说明：以下路径均需加上统一前缀 `/api`。

### 3.1 认证接口

| 方法 | 路径 | 作用 | 说明 |
| --- | --- | --- | --- |
| POST | `/user/login` | 用户登录 | 入参 `username`、`password`，返回 token、用户信息、角色列表 |
| POST | `/user/logout` | 用户登出 | 读取 `Authorization: Bearer xxx`，将 token 拉黑 |
| GET | `/user/info` | 获取当前登录用户信息 | 从 token 中解析用户 ID |
| GET | `/user/menu/tree` | 获取当前用户菜单树 | 当前实现返回空列表 |
| GET | `/user/token/validate` | 校验 token 是否有效 | 返回 `true/false` |
| PUT | `/user/update` | 修改当前登录人个人资料 | 修改 `realName`、`email`、`phone` |
| POST | `/user/change-password` | 修改当前登录人密码 | 入参 `oldPassword`、`newPassword` |

### 3.2 用户管理接口

| 方法 | 路径 | 作用 | 权限 |
| --- | --- | --- | --- |
| GET | `/user/list` | 用户分页列表 | 管理员 |
| GET | `/user/page` | 用户分页列表 | 管理员 |
| GET | `/user/query` | 用户分页列表，附带角色信息 | 管理员 |
| GET | `/user/{id}` | 查询用户基础信息 | 无 `@RequireAdmin` |
| GET | `/user/{id}/detail` | 查询用户详情及角色 | 管理员 |
| POST | `/user` | 新增用户 | 管理员 |
| PUT | `/user` | 修改用户 | 管理员 |
| PUT | `/user/{id}` | 按 ID 修改用户 | 管理员 |
| DELETE | `/user/{id}` | 删除用户 | 管理员 |
| PUT | `/user/resetPwd` | 重置用户密码 | 管理员 |
| GET | `/user/checkUsername` | 检查用户名唯一性 | 无 `@RequireAdmin` |
| PUT | `/user/{id}/status` | 启用/禁用切换 | 管理员 |

用户管理主要请求参数：

- `POST /user`

  - `phone`
  - `status`
  - `deptId`
  - `roleId` 或 `roleIds`
- `PUT /user` / `PUT /user/{id}`
  - `id`
  - `realName`
  - `email`
  - `phone`
  - `status`
  - `deptId`
  - `roleId` 或 `roleIds`
- `PUT /user/resetPwd`
  - `userId`
  - `newPassword`

## 6. 当前实现中的几个重要说明

- 认证模块的 `/user/menu/tree` 还没有真正查询菜单，当前直接返回空数组。
- `/user/info` 返回的角色信息是按用户名是否为 `admin` 做简化构造，不是完整 RBAC 明细。
- 管理员校验依赖 `super_admin` 角色编码。
- 多数实体启用了 MyBatis-Plus 逻辑删除，但部分自定义 SQL 仍直接访问物理表。
- XML 中有对 `sys_user_role`、`sys_role_menu` 的 `deleted` 字段过滤，但初始化 SQL 并没有给这两张关联表定义 `deleted` 字段；如果这些 SQL 被执行，数据库结构需要进一步统一。
- 用户删除、角色删除当前使用的是物理删除方法。
